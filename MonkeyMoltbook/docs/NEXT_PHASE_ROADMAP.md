# MonkeyMoltbook — Next-Phase Roadmap

Updated: 2026-03-23 America/Los_Angeles

## Purpose

Define what matters next now that the core product skeleton exists and the Moltbook execution surface is live.

This is the roadmap for the next phase after initial MVP construction.

---

## Current truth

MonkeyMoltbook now already has:
- mobile-first swipe shell
- local agent roster
- swipe flow
- preload queue
- hook validation
- session-limit shell
- response quality system
- Moltbook integration
- public-feed-derived Moltbook candidate ingestion
- real claimed Moltbook account (`moeylarge`)
- first live Moltbook posts/comments/follows already executed

So the next phase is **not** primarily scaffolding.

It is:
- participation
- signal collection
- quality refinement
- feedback-loop building

---

# Next-phase thesis

The next phase of MonkeyMoltbook is:

## Post → interact → observe → ingest → refine

That is the loop.

Not a spam loop.
A product-learning loop.

---

# Primary workstreams

## 1. Moltbook execution
Goal:
- become visible
- attract strong replies
- build a high-quality source graph

What this includes:
- thoughtful posting
- selective commenting
- selective following
- replies when warranted
- no autopost loops
- strict security policy adherence

Success signals:
- more replies
- more follows from strong accounts
- more useful conversations
- better high-signal source accounts entering orbit

---

## 2. Candidate-source development
Goal:
- improve the Moltbook secondary pool
- discover stronger agent voices
- reduce weak/fallback candidates

What this includes:
- tracking high-signal accounts
- ranking accounts by usefulness
- identifying which post/comment themes surface the best candidates
- refining admission filters over time

Success signals:
- fewer fallback hooks
- stronger derived archetypes
- more memorable Moltbook-derived candidates

---

## 3. Derivation quality refinement
Goal:
- make public-feed-derived Moltbook candidates better

What this includes:
- better style classification
- better archetype compression
- stronger hook generation
- better rejection of noisy candidates

Success signals:
- lower fallback-hook rate
- stronger hook validation pass rate
- better alignment between source voice and swipe-agent behavior

---

## 4. Measurement / logging
Goal:
- stop guessing
- measure what actually works

What this includes:
- tracking post/comment outcomes
- tracking discovered candidates
- tracking which source accounts become admitted agents
- tracking which interaction patterns lead to better source quality

Success signals:
- visible evidence of which Moltbook behaviors produce the best downstream product value

---

# What is not the main bottleneck right now

Not the main bottleneck now:
- more UI screens
- more product features
- more architectural expansion
- billing integration before signal quality improves

The main bottleneck now is:
- quality of participation
- quality of source data
- quality of derived agents

---

# Operating rules

## Do
- post deliberately
- comment more than you post early on
- follow selectively
- use Moltbook to build presence and source quality
- keep local agents as the stable primary layer

## Do not
- post for volume alone
- create loops
- confuse activity with traction
- ingest noisy accounts blindly
- let Moltbook participation reduce product quality or security discipline

---

# Recommended weekly rhythm

## Daily
- 1–3 strong posts max
- several thoughtful comments/replies
- selective follows
- update source tracker when strong accounts appear

## Every few days
- review which Moltbook interactions produced useful candidates
- tighten derivation rules
- refine posting themes

## Weekly
- review traction
- review source quality
- review candidate admission quality
- review whether Moltbook activity is actually improving MonkeyMoltbook

---

# Concrete next priorities

## Priority 1
Continue Moltbook execution under the security policy.

## Priority 2
Track strong accounts and improve the public-feed-derived candidate pool.

## Priority 3
Add light logging/analytics so outcomes are measurable.

## Priority 4
Only after signal quality improves, revisit deeper monetization or bigger product expansion.
